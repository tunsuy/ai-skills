# 方法论提炼指南

> 本指南帮助 Meta Creator 在信息收集阶段，从用户的模糊描述中提炼出结构化的方法论。

---

## 1. 提炼框架：PASDO 模型

每个方法论都可以用以下五个维度来描述：

| 维度 | 含义 | 关键问题 |
|------|------|----------|
| **P**urpose | 目的 | 这个流程最终要达成什么？ |
| **A**ctors | 角色 | 谁参与？各自职责是什么？ |
| **S**tages | 阶段 | 从开始到完成经历哪些阶段？ |
| **D**eliverables | 产出 | 每个阶段和最终交付什么？ |
| **O**bservables | 可观测性 | 如何判断做得好不好？ |

---

## 2. 阶段提炼技巧

### 2.1 从用户叙事中提取阶段

用户通常用叙事方式描述流程，需要从中识别"状态转换"：

**用户说**："我一般先看看竞品怎么做的，然后想想自己的方案，写个文档，让团队看看，改改就交付了。"

**提取过程**：
1. "看看竞品" → **调研阶段**
2. "想想自己的方案" → **设计阶段**
3. "写个文档" → **文档化阶段**
4. "让团队看看" → **评审阶段**
5. "改改就交付" → **修订+交付阶段**

### 2.2 阶段粒度判断

- **太粗**（1-2个阶段）：问"这个步骤里面，有没有可以分开做的子步骤？"
- **太细**（8+个阶段）：问"这些步骤中，哪些可以合并成一个环节？"
- **刚好**（3-6个阶段）：适合大多数领域

### 2.3 常见遗漏阶段

用户常常忘记描述这些阶段，需要主动引导：

| 遗漏类型 | 典型场景 | 引导问题 |
|----------|----------|----------|
| 前置准备 | 直接开始做事，忽略需求定义 | "在开始之前，你通常需要先弄清楚什么？" |
| 复盘总结 | 做完就走，不沉淀经验 | "做完之后，你会回顾一下过程吗？" |
| 验证反馈 | 产出后直接交付，不验证 | "怎么确认做出来的东西是对的？" |
| 风险评估 | 全程乐观推进，忽略风险 | "这个过程中，最容易出问题的环节是什么？" |

## 3. 流程类型识别

### 3.1 判断依据

| 信号 | 推荐流程类型 |
|------|-------------|
| "一步步来"、"先...再..." | `linear` |
| "做完要检查"、"需要评审通过" | `linear_with_gate` |
| "可能要返工"、"做着做着发现不对" | `directed_with_backtrack` |
| "几个方面同时推进"、"多线并行" | `parallel_convergent` |

### 3.2 混合流程处理

现实中很多领域是混合的，处理策略：

1. **识别主干流程类型**：选最能代表整体特征的
2. **在特定阶段嵌入子流程**：如主干是 linear，但某个阶段内部有回退逻辑
3. **不要过度复杂化**：如果混合程度高，优先选 `directed_with_backtrack`（它是最灵活的）

## 4. 记忆锚点识别

### 4.1 什么值得记住？

用"事后回溯测试"来判断：假设 3 个月后回头看这个项目——

| 问题 | 对应记忆类型 |
|------|-------------|
| "当初为什么选了方案A？" | `decisions.md` |
| "进度到哪了？" | `plan.md` |
| "之前遇到了什么坑？" | `issues.md` / `risks.md` |
| "关键发现是什么？" | `findings.md` / `insights.md` |
| "最终产出了什么？" | `output/` 目录 |

### 4.2 记忆文件设计原则

1. **单一职责**：一个文件只记一类信息
2. **追加优先**：decisions 和 issues 类文件只追加，不修改
3. **覆盖更新**：plan 和 status 类文件可覆盖，但变更记录在 decisions 中
4. **可读性**：文件格式对人类友好（Markdown），不用 JSON/YAML

## 5. 双模式设计指南

### 5.1 如何确定模式分界线

每个领域需要找到一个"模式分界维度"：

| 领域 | 分界维度 | 轻量阈值 | 完整阈值 |
|------|----------|----------|----------|
| 学习 | 时间跨度 | ≤2周 | ≥1个月 |
| 设计 | 项目范围 | 单功能迭代 | 0→1新产品 |
| 构建 | 项目规模 | 单一交付物 | 多模块系统 |
| 写作 | 篇幅 | 单篇文章 | 系列/书籍 |
| 研究 | 深度 | 快速调研 | 深度研究 |

### 5.2 模式差异化维度

确定分界线后，在以下维度上定义差异：
- 阶段数量（轻量少、完整多）
- 分析深度（轻量快速、完整深入）
- 复盘频率（轻量按需、完整每阶段）
- 文件管理（轻量全量、完整分层）
- 结项产出（轻量精简、完整完整）

## 6. 常见反模式

### ❌ 不要做的事

| 反模式 | 问题 | 正确做法 |
|--------|------|----------|
| 收集所有可能的信息 | 用户疲劳、放弃使用 | 只收集必要信息，其余被动捕获 |
| 流程过于复杂 | 用户无法理解和遵循 | 主流程 ≤ 6 个阶段 |
| 没有跳出机制 | 用户被困在流程中 | 每个阶段允许"标记跳过" |
| Memory 文件过多 | 上下文窗口被占满 | 核心文件 ≤ 5 个 |
| 只有一种模式 | 杀鸡用牛刀或大材小用 | 必须有轻量/完整双模式 |
