# 流程模式参考

> 本文件定义了 Meta Creator 支持的四种流程模式，以及对应的生成规则。
> 每种模式是一个"流程骨架"，用于生成 Creator 内部的阶段执行逻辑。

---

## 模式总览

```
┌─────────────────────────────────────────────────────────────┐
│                      四种流程模式                             │
├────────────────┬────────────────────────────────────────────┤
│ linear         │ A → B → C → D → End                       │
│ linear_gate    │ A →[✓]→ B →[✓]→ C →[✓]→ D → End          │
│ backtrack      │ A →[✓]→ B →[✓]→ C ←──[✗]── D             │
│ parallel       │ ┌A─┐                                       │
│                │ ├B─┤→ Merge → End                          │
│                │ └C─┘                                       │
└────────────────┴────────────────────────────────────────────┘
```

## 模式 1: Linear（线性流程）

### 适用领域
学习、培训、入职引导、习惯养成

### 特征
- 阶段严格有序
- 每个阶段完成后自动进入下一阶段
- 不存在回退机制
- 简单、清晰、易于理解

### 生成规则

```markdown
## 阶段执行

当前阶段：{{current_phase}}

### 阶段 {N}: {phase_name}

**目标**：{phase_goal}

**执行步骤**：
1. {action_1}
2. {action_2}
3. ...

**产出物**：{phase_output}
→ 保存到 `{phase_dir}/{output_file}`

**完成条件**：{phase_gate}

完成后：
- 更新 plan.md 中的进度标记
- 自动进入阶段 {N+1}
```

### 启动协议模板

```markdown
## 启动协议

1. 读取 plan.md
2. 识别当前阶段（找到第一个未标记 ✅ 的阶段）
3. 声明："当前在阶段 {N}: {name}，上次完成了 {last_action}"
4. 继续推进
```

## 模式 2: Linear with Gate（线性+门控）

### 适用领域
设计项目、写作项目、方案制定、旅行规划

### 特征
- 阶段有序，但需要通过"门控"才能进入下一阶段
- 门控可以是：自检清单、质量标准、外部确认
- 门控未通过时，停留在当前阶段继续完善
- 比纯线性多了质量保障

### 生成规则

```markdown
## 阶段执行

当前阶段：{{current_phase}}

### 阶段 {N}: {phase_name}

**目标**：{phase_goal}

**执行步骤**：
1. {action_1}
2. {action_2}
3. ...

**产出物**：{phase_output}
→ 保存到 `{phase_dir}/{output_file}`

**门控检查**：
进入下一阶段前必须确认：
- [ ] {gate_check_1}
- [ ] {gate_check_2}
- [ ] {gate_check_3}

门控通过 → 更新 plan.md，进入阶段 {N+1}
门控未通过 → 列出未满足项，继续完善当前阶段
```

### 门控设计原则

1. **可检验**：每项门控必须是"是/否"问题
2. **不超过 5 项**：太多会导致阶段间流动停滞
3. **区分硬性/软性**：
   - 硬性门控（必须通过）：`[MUST]` 标记
   - 软性门控（建议通过）：`[SHOULD]` 标记
4. **记录门控结果**：门控通过/未通过的记录写入 decisions.md

## 模式 3: Directed with Backtrack（有向可回退）

### 适用领域
工程开发、科研实验、产品迭代、健康管理

### 特征
- 主流程有方向性
- 允许在特定条件下回退到之前的阶段
- 回退是受控的（有明确的触发条件和目标阶段）
- 最灵活但也最复杂

### 生成规则

```markdown
## 阶段执行

当前阶段：{{current_phase}}
回退次数：{{backtrack_count}}

### 阶段 {N}: {phase_name}

**目标**：{phase_goal}

**执行步骤**：
1. {action_1}
2. {action_2}
3. ...

**产出物**：{phase_output}
→ 保存到 `{phase_dir}/{output_file}`

**前进条件**：{forward_gate}
**回退触发**：
- 条件：{backtrack_condition}
- 目标阶段：阶段 {M}
- 回退动作：
  1. 在 decisions.md 记录回退原因
  2. 将当前阶段产出标记为"待修订"
  3. 回到阶段 {M} 的 {specific_step}
  4. 更新 plan.md 中的回退标记
```

### 回退规则设计原则

1. **回退有理由**：每次回退必须在 decisions.md 中记录原因
2. **回退有目标**：不是"回到开头重来"，而是回到特定阶段的特定步骤
3. **回退有限制**：同一回退路径最多触发 3 次，超过则升级为"方案重评"
4. **保留产出**：回退不删除已有产出，而是标记为"待修订"
5. **记录回退次数**：在 plan.md 中追踪总回退次数，用于结项时的效率分析

### 回退路径设计模板

```
常见回退路径（用于生成时参考）：

验证 → 实施    : "测试不通过，需要修改实现"
验证 → 设计    : "测试发现设计缺陷，需要修改方案"
实施 → 技术决策 : "实现过程中发现技术方案不可行"
设计 → 需求    : "设计过程中发现需求理解有偏差"
```

## 模式 4: Parallel Convergent（并行后汇聚）

### 适用领域
投资研究、多维调研、团队协作、多源数据分析

### 特征
- 存在可以独立推进的"分支"
- 分支之间可能有弱依赖（共享部分信息）
- 在"汇聚点"合并所有分支的成果
- 需要额外的进度可视化

### 生成规则

```markdown
## 阶段执行

### 启动阶段
**目标**：定义分支和汇聚条件
**产出**：
- 分支定义（每个分支的目标和预期产出）
- 汇聚条件（全部完成 / N-of-M 完成）
- 依赖关系图

### 并行分支

#### 分支 {A}: {branch_name}
**目标**：{branch_goal}
**状态**：{{branch_status}} (pending / in_progress / completed)
**执行步骤**：
1. {action_1}
2. {action_2}
**产出物**：{branch_output}
→ 保存到 `branch_{A}/{output_file}`

#### 分支 {B}: {branch_name}
**目标**：{branch_goal}
**状态**：{{branch_status}}
...

### 进度可视化

| 分支 | 状态 | 完成度 | 阻塞 |
|------|------|--------|------|
| A    | ██░░ | 50%    | 无   |
| B    | ███░ | 75%    | 无   |
| C    | █░░░ | 25%    | 等待外部数据 |

### 汇聚阶段
**触发条件**：{convergence_condition}
**汇聚动作**：
1. 读取所有分支的产出物
2. 交叉分析和综合
3. 识别分支间的冲突或互补
4. 生成综合产出
**产出物**：{merged_output}
```

### 并行规则设计原则

1. **分支独立性**：每个分支必须能独立推进，不等待其他分支
2. **弱依赖处理**：如果分支间有信息共享需求，通过 `shared/` 目录传递
3. **灵活汇聚**：支持"全部完成后汇聚"和"N/M完成即汇聚"两种模式
4. **进度可视化**：在 plan.md 中用表格展示各分支进度
5. **阻塞标记**：分支遇到外部阻塞时标记，不影响其他分支推进

## 模式选择决策树

```
用户描述的流程中：

有并行的独立工作线？
├─ 是 → parallel_convergent
└─ 否 →
    做的过程中可能需要返工？
    ├─ 是 → directed_with_backtrack
    └─ 否 →
        阶段之间需要质量检查/评审？
        ├─ 是 → linear_with_gate
        └─ 否 → linear
```

## 混合模式处理

现实中很多领域是混合的，处理策略：

### 策略 1: 主干+嵌套

选择主干流程类型，在特定阶段内部嵌入子流程：

```
主干：linear_with_gate
  阶段 3（开发）内部：directed_with_backtrack
    设计 → 实现 → 测试 ←[回退]── 测试
```

### 策略 2: 分段

不同阶段组使用不同模式：

```
阶段 1-2：linear（准备阶段）
阶段 3-4：parallel_convergent（执行阶段）
阶段 5：linear_with_gate（交付阶段）
```

### 策略 3: 降级为最灵活模式

如果混合程度太高，直接使用 `directed_with_backtrack`，
它是最灵活的模式，可以模拟其他三种模式的行为。
