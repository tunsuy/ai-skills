# Memory 系统设计模式

> 本文件定义了 Meta Creator 生成的 Creator 中，Memory 系统的设计模式和最佳实践。
> Memory 系统是整套 Creator 体系的核心差异化价值。

---

## 1. 设计哲学

### 核心理念
**文件即记忆，目录即结构，git 即版本**

- 不依赖数据库、不依赖平台 API
- 用户可以直接查看、编辑所有记忆文件
- 可以用 git 管理版本历史
- AI 通过读取文件恢复上下文，通过写入文件保存记忆

### 三条铁律

1. **只增不减**：decisions.md 等追溯类文件只追加，永远不修改历史记录
2. **覆盖有痕**：plan.md 等状态类文件可覆盖更新，但变更原因记录在 decisions.md
3. **严禁删除**："归档"仅意味着调整读取优先级，文件本身永远保留

---

## 2. 文件分类体系

### 2.1 核心文件（每个 Creator 都必须生成）

| 文件 | 类型 | 写入模式 | 读取优先级 |
|------|------|----------|-----------|
| `plan.md` | 状态型 | 覆盖更新 | P0（每次必读） |
| `decisions.md` | 追溯型 | 追加写入 | P0（最近 N 条） |
| `file_structure.md` | 索引型 | 覆盖更新 | P0（每次必读） |

#### plan.md 标准格式

```markdown
# {项目名称} — 项目计划

## 基本信息
- 创建时间：{timestamp}
- 模式：{轻量/完整}
- 当前阶段：{current_phase}
- 总进度：{N}/{total} 阶段完成

## 阶段计划

### 阶段 1: {name} ✅
- 开始：{date}
- 完成：{date}
- 产出：{output_files}

### 阶段 2: {name} 🔄 ← 当前
- 开始：{date}
- 进度：{description}

### 阶段 3: {name} ⏳
- 预计开始：{date}
```

#### decisions.md 标准格式

```markdown
# 决策记录

> 本文件记录所有关键决策，只追加不修改。

---

## [D001] {决策标题} — {date}

**背景**：{什么情况下需要做这个决策}

**选项**：
1. {选项A} — {优点} / {缺点}
2. {选项B} — {优点} / {缺点}
3. {选项C} — {优点} / {缺点}

**决定**：选择 {选项X}

**理由**：{为什么选择这个}

**影响**：{这个决策影响哪些后续步骤}

---

## [D002] ...
```

### 2.2 领域特定文件

根据领域不同，选择合适的记忆文件类型：

| 记忆类型 | 适用领域 | 文件名建议 | 写入模式 |
|----------|----------|-----------|----------|
| 学习笔记 | 学习、培训 | `notes_{topic}.md` | 创建不修改 |
| 研究发现 | 研究、投资 | `findings.md` | 追加写入 |
| 风险登记 | 工程、项目管理 | `risks.md` | 追加+状态更新 |
| 实验日志 | 科研、测试 | `experiment_log.md` | 追加写入 |
| 草稿 | 写作、设计 | `drafts/{name}.md` | 覆盖更新 |
| 反馈记录 | 设计、产品 | `feedback.md` | 追加写入 |
| 里程碑 | 职业发展 | `milestones.md` | 追加写入 |
| 检查结果 | 健康、质量 | `check_results.md` | 追加写入 |

### 2.3 阶段产出文件

每个阶段的产出物存放在对应的阶段目录下：

```
project/
├── phase_1_{name}/
│   ├── {output_1}.md
│   ├── {output_2}.md
│   └── summary.md        # 阶段总结（完成时生成）
├── phase_2_{name}/
│   └── ...
```

## 3. 读取优先级策略

### 3.1 轻量模式：全量加载

```
读取所有文件，不做优先级区分。
适用条件：总文件数 ≤ 10 且总字符数 ≤ 50K
```

### 3.2 完整模式：分层优先级

```
P0 (每次必读) ───────────────────────────
  plan.md（完整）
  decisions.md（最近 5 条）
  file_structure.md（完整）

P1 (当前阶段) ───────────────────────────
  当前阶段目录下所有文件

P2 (相邻阶段) ───────────────────────────
  前一阶段的 summary.md
  后一阶段的计划/设计文件（如有）

P3 (按需加载) ───────────────────────────
  更早阶段的文件
  仅在用户明确提及或当前任务需要时加载
```

### 3.3 智能加载触发规则

| 触发条件 | 动作 |
|----------|------|
| 用户提到某个历史阶段的内容 | 加载该阶段的 summary 和相关文件 |
| 当前决策与历史决策有关联 | 加载 decisions.md 的相关条目 |
| 进入新阶段 | 卸载 P3 文件，加载新阶段的 P1 文件 |
| 上下文窗口接近上限 | 将 P2 降级为 P3，仅保留 P0 和 P1 |

## 4. 启动协议（跨对话恢复）

每个生成的 Skill 必须包含以下启动逻辑：

```
启动协议：
1. 检查项目目录是否存在
   - 不存在 → 首次启动，进入初始化流程
   - 存在 → 继续项目，进入恢复流程

2. 恢复流程：
   a. 读取 plan.md → 获取当前阶段和进度
   b. 读取 decisions.md（最近 5 条） → 获取最新决策上下文
   c. 读取当前阶段的文件 → 获取工作细节
   d. 声明恢复结果：
      "欢迎回来！当前项目状态：
       - 阶段：{current_phase}
       - 上次进展：{last_progress}
       - 下一步：{next_action}
       有什么需要调整的吗？还是继续推进？"

3. 异常处理：
   - plan.md 损坏/缺失 → 扫描目录结构推断进度
   - decisions.md 为空 → 正常，表示还没有关键决策
   - 阶段目录不完整 → 提示用户确认，可能是手动修改过
```

## 5. 写入规范

### 5.1 时间戳格式

所有文件中的时间戳使用 ISO 8601 格式：
```
2026-04-19
2026-04-19T11:30
```

### 5.2 Markdown 格式规范

- 标题层级：`#` 文件标题、`##` 主要分节、`###` 子节
- 列表：用 `-` 而非 `*`
- 代码块：用三反引号 + 语言标识
- 分隔线：用 `---`

### 5.3 文件大小控制

| 文件类型 | 建议上限 | 超限处理 |
|----------|----------|----------|
| plan.md | 5KB | 压缩历史阶段描述 |
| decisions.md | 20KB | 分卷（decisions_archive_1.md） |
| 阶段笔记 | 10KB/文件 | 拆分为多个子文件 |
| 阶段总结 | 3KB | 保持精炼 |

---

## 6. 跨 Creator 桥接（handoff 协议）

当一个 Creator 的项目结项时，自动生成 `handoff.md` 供下游 Creator 消费：

### handoff.md 标准格式

```markdown
# Handoff: {项目名称}

## 来源
- Creator：{creator_name}
- 项目：{project_name}
- 完成日期：{date}

## 关键产出
{最终产出物清单及摘要}

## 核心决策
{从 decisions.md 中提取的关键决策摘要}

## 下游建议
{对使用这些产出的下一个 Creator/流程的建议}

## 已知风险/限制
{需要下游注意的风险和约束}
```
